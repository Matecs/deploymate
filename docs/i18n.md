# Internationalization (i18n) Guide

DeployMate supports **English** (`en`) and **Hungarian** (`hu`). This document explains how the i18n system works and how to add or modify translations.

## How It Works

All i18n logic lives in a single file: **`src/lib/i18n.tsx`**.

The file exports:

| Export | Type | Description |
|---|---|---|
| `Lang` | `type` | `"en" \| "hu"` |
| `LangProvider` | React component | Context provider — wrap your page tree with this |
| `useLang` | React hook | Access language state and translations inside any component |

The `LangProvider` reads the saved language preference from `localStorage` on mount and falls back to `"en"`. Whenever the user switches the language, the new preference is written back to `localStorage` so it persists across sessions.

---

## Translation Dictionary

Translations are stored as a plain TypeScript `const` object called `t` inside `src/lib/i18n.tsx`. Each key maps to an object with `en` and `hu` string values:

```ts
const t = {
  "hero.title": { en: "QA-Driven Release & Compliance Operations", hu: "QA-Driven Release & Compliance Operations" },
  "hero.cta":   { en: "Book a 15-min call in my Google Calendar", hu: "Foglalj 15 perces hívást a Google Calendar-omban" },
  // ...
} as const;
```

Using `as const` means TypeScript infers the exact literal types for every key and value, which provides:

- **Exhaustiveness checking** — you cannot call `t("unknown.key")` without a type error.
- **Both-language requirement** — every key must have both `en` and `hu` values.

---

## Key Naming Convention

Keys follow the pattern `<section>.<item>`:

| Prefix | Section |
|---|---|
| `nav.*` | Header navigation |
| `hero.*` | Hero section |
| `pain.*` | Pain points section |
| `audience.*` | Audience / "Who is this for?" section |
| `packages.*` | Service packages section |
| `howItWorks.*` | How it works section |
| `cred.*` | Credibility / testimonials section |
| `cta.*` | Call-to-action section |
| `footer.*` | Footer |

Sub-keys use descriptive names: `.title`, `.desc`, `.tag`, `.price`, etc.

## Using Translations in Components

1. Import `useLang` and call it inside your component.
2. Destructure `t` (the translate function), and optionally `lang` (current language code) and `setLang` (setter).

```tsx
import { useLang } from "@/lib/i18n";

const MySection = () => {
  const { t } = useLang();

  return (
    <section>
      <h2>{t("hero.title")}</h2>
      <p>{t("hero.desc")}</p>
    </section>
  );
};
```

`t(key)` returns the string for the currently active language. If the key is not found, it returns the key itself as a fallback.

> **Note:** `useLang()` must be called inside a component that is a descendant of `<LangProvider>`. If it is called outside, it will throw an error.

## Language Switcher

The language toggle in `Header.tsx` uses `setLang` to flip between the two languages:

```tsx
const { lang, setLang } = useLang();

<button onClick={() => setLang(lang === "en" ? "hu" : "en")}>
  {lang === "en" ? "HU 🇭🇺" : "EN 🇬🇧"}
</button>
```

## Adding a New Translation Key

1. Open `src/lib/i18n.tsx`.
2. Add a new entry to the `t` object with values for **both** `en` and `hu`:

```ts
"mySection.newKey": {
  en: "English text here",
  hu: "Magyar szöveg itt",
},
```

3. Use the key in your component:

```tsx
const { t } = useLang();
// ...
<p>{t("mySection.newKey")}</p>
```

TypeScript will raise a compile error if you forget to provide either the `en` or `hu` value, or if you try to use a key that does not exist in the dictionary.

## Adding a New Language

The system is designed for two languages, but extending it to a third is straightforward:

1. **Update the `Lang` type** in `src/lib/i18n.tsx`:

```ts
export type Lang = "en" | "hu" | "de"; // add new locale code
```

2. **Add the new locale to every key** in the `t` object:

```ts
"hero.title": { en: "...", hu: "...", de: "..." },
```

3. **Update the `LangProvider`** initial state logic to handle the new locale code from `localStorage`.

4. **Update the language switcher** in `Header.tsx` to cycle through (or select from) the available languages.

5. Verify there are no TypeScript errors with `npm run lint` or `npx tsc --noEmit`.
